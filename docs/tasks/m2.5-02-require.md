## What this task does

Adds `require` statements â€” runtime assertions that validate a boolean expression at
the point of execution. Supports an optional message, an optional warn-only mode that
emits a diagnostic instead of halting, and an optional `set` recovery closure that is
called once before re-evaluating a failing check.

---

## Syntax

```rust
// Basic â€” halts with RequireError if false
require x > 0

// With message
require x > 0, "x must be positive"

// Warn mode â€” emits warning, does not halt
require(warn) x > 0, "x should be positive"

// With set recovery â€” call set(), retry once, then fail/warn
require x > 0, set: || { x = default_value() }

// Warn + set
require(warn) x > 0, "x should be positive", set: || { x = default_value() }
```

## ferric_common â€” additions required

```rust
pub struct RequireStmt {
    pub span:    Span,
    pub mode:    RequireMode,
    pub expr:    Box<Expr>,
    pub message: Option<Box<Expr>>,   // must resolve to Str at runtime
    pub set_fn:  Option<Box<Expr>>,   // must resolve to Fn() -> Unit
}

pub enum RequireMode {
    Error,   // default â€” halts on failure
    Warn,    // emits diagnostic, continues
}

pub struct RequireError {
    pub span:    Span,
    pub mode:    RequireMode,
    pub message: Option<String>,   // resolved at runtime; None if no message provided
}
```

`RequireError` carries a `Span` (Rule 5).

---

## Execution semantics

When a `require` statement is evaluated by the VM:

1. Evaluate `expr`. If it is `true`, continue normally â€” done.
2. If `false` and a `set_fn` is present:
   a. Call `set_fn()` (zero arguments, return value discarded).
   b. Re-evaluate `expr` once.
   c. If now `true`, continue normally â€” done.
3. If still `false` (or no `set_fn`):
   - `RequireMode::Error` â€” propagate `RequireError` up the call stack. Execution halts.
   - `RequireMode::Warn`  â€” render a warning through the diagnostics renderer and continue.

The `set_fn` closure may mutate variables in its capture scope. It may not be async.
It is called at most once per `require` evaluation â€” there is no retry loop.

`require` does not return a value. It is a statement only.

## Stage changes

### Lexer

Add `Require` as a reserved keyword token.

`Warn` and `Set` are **contextual keywords** â€” lex them as `Ident` and let the parser
promote them to keyword meaning only in `require` context. This avoids reserving
`warn` and `set` as identifiers in user code.

```rust
// Add to token enum:
Require,    // `require`
```

### Parser

Parse `require` statements into `RequireStmt`. The grammar is:

```
require_stmt ::= "require" ("(" "warn" ")")? expr
                 ("," string_expr)?
                 ("," "set" ":" closure_expr)?
```

- `"warn"` inside `require(...)` is matched as an identifier with value `"warn"`.
  Any other identifier or expression here is a `ParseError::InvalidRequireMode`.
- The message (if present) is any expression â€” it is not required to be a string
  literal. It will be type-checked as `Str` later.
- The `set:` label is matched as an identifier with value `"set"` followed by `:`.
  This is the same `name: value` shape as named function arguments.

**New error variant:**

```rust
// Add to ParseError:
ParseError::InvalidRequireMode { span: Span }   // anything other than `warn` in require(...)
```

Carry `Span` (Rule 5).

### Name Resolution

Resolve `expr` and `message` as normal expressions in the current scope.

For `set_fn`: resolve as a closure expression. Check that the closure has zero
declared parameters. If it has parameters, emit `ResolveError::RequireSetArity { span: Span }`.

```rust
// Add to ResolveError:
ResolveError::RequireSetArity { span: Span }   // set closure must take zero arguments
```

Carry `Span` (Rule 5).

### Type Checker

Three checks:

1. `expr` must be `Bool`. If not, emit `TypeError::RequireNonBool { found: Ty, span: Span }`.
2. `message` (if present) must be `Str`. If not, emit `TypeError::RequireMessageNonStr { found: Ty, span: Span }`.
3. `set_fn` (if present) must be `Fn() -> Unit`. If not, emit `TypeError::RequireSetType { found: Ty, span: Span }`.

```rust
// Add to TypeError:
TypeError::RequireNonBool      { found: Ty, span: Span }
TypeError::RequireMessageNonStr { found: Ty, span: Span }
TypeError::RequireSetType      { found: Ty, span: Span }
```

All carry `Span` (Rule 5).

### VM (TreeWalker)

Evaluate `RequireStmt` per the execution semantics above. On failure in
`RequireMode::Error`, propagate `RequireError` as a runtime error (same mechanism
as other runtime errors â€” it surfaces to `main.rs` which prints and exits 1).

The `message` expression is only evaluated if the check fails, and only if a message
was provided. Do not eagerly evaluate it.

### Diagnostics

Add a rendering arm for `RequireError`. No structural change to the renderer â€”
it already handles errors with `Span`. The output format should match M2's
span-annotated style:

```
error: require failed: x must be positive
  --> main.fe:12:5
   |
12 |     require x > 0, "x must be positive"
   |     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ condition evaluated to false
```

For warn mode, render using the existing warning path (or add one if it doesn't
exist yet â€” same shape as errors but prefixed `warning:` instead of `error:`).
