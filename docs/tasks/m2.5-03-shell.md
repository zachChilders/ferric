# M2.5 — Task 3: `$` Shell Expressions

> **Prerequisite:** Task 1 (Named Parameters) must be complete and all tests passing
> before starting this task. Tasks 2 and 3 are independent of each other and may run
> concurrently if two agents are available.

---

## What this task does

Adds `$` shell expressions — a first-class syntax for running subprocess commands
and capturing their output as a typed value. Commands are newline-terminated,
support `\` continuation for multi-line commands, and use `@{expr}` for splicing
Ferric values into the command string.

---

## Syntax

```
// Basic — runs to end of line
let result = $ git rev-parse HEAD

// @{expr} interpolation — only Ferric interpolation syntax; everything else is literal
let count = $ cat @{filename} | wc -l

// Literal shell characters pass through untouched
$ awk '{print $1}' @{dir}/file.txt

// Discard output
$ mkdir -p @{build_dir}

// Multi-line via \ continuation
let log = $ git log \
              --oneline \
              --decorate
```

A `$` expression is an expression, not a statement. It can appear anywhere an
expression is valid: RHS of a `let`, argument to a function, condition in an `if`.

---

## ferric_common — additions required

```rust
pub struct ShellExpr {
    pub span:  Span,
    pub parts: Vec<ShellPart>,
}

pub enum ShellPart {
    Literal(String),
    Interpolated(Box<Expr>),   // the expr inside @{...}
}

pub struct ShellOutput {
    pub stdout:    String,
    pub exit_code: i32,
}
```

`ShellOutput` is a new value type. It is **not** a `Str` — callers must access
`.stdout` and `.exit_code` explicitly. This is intentional: swallowing exit codes
silently is a common source of bugs.

---

## Lexer — the main work of this task

The lexer is responsible for all shell-line recognition. By the time a `ShellLine`
token reaches the parser, all interpolation has been pre-split and the Ferric token
stream inside each `@{...}` has been lexed normally. The parser never sees raw shell
text.

### Recognition

A `$` token at the start of an expression position (i.e. where an expression can
begin, not inside an expression already in progress) switches the lexer into
**shell-line mode**.

In shell-line mode:

- Consume characters to end of line.
- If the last non-whitespace character before the newline is `\`, strip it,
  consume the newline, and continue consuming the next line (repeat as needed).
  Leading whitespace on continuation lines is preserved.
- Split the accumulated raw text on `@{` / `}` boundaries to produce `ShellPart`
  segments.
- For each `Interpolated` segment, recursively lex the content as a Ferric
  token stream.
- Emit a single composite `ShellLine(Vec<ShellPart>)` token.

### Interpolation splitting rules

`@{` begins an interpolated segment. `}` ends it. Everything else — including `{`,
`}` not preceded by `@`, `$`, pipes, quotes, and all other shell metacharacters —
is treated as a literal character and included verbatim in the nearest `Literal`
segment.

```
"ls -la @{dir}/tmp"
→ [ Literal("ls -la "), Interpolated([Ident("dir")]), Literal("/tmp") ]

"awk '{print $1}' @{dir}/file"
→ [ Literal("awk '{print $1}' "), Interpolated([Ident("dir")]), Literal("/file") ]
```

Nested `@{` inside an interpolated segment is a `LexError::NestedShellInterp`.

An `@{` without a closing `}` before end-of-line is a `LexError::UnclosedShellInterp`.

### New token

```rust
// Add to token enum:
ShellLine(Vec<ShellPart>),
```

### New error variants

```rust
// Add to LexError:
LexError::NestedShellInterp  { span: Span }   // @{ inside @{...}
LexError::UnclosedShellInterp { span: Span }  // @{ without closing }
```

Both carry `Span` (Rule 5).

## Parser

Parse `ShellLine(parts)` into `Expr::Shell(ShellExpr { span, parts })`. The parts
are already fully resolved by the lexer — the parser only wraps them in the AST node.

No new error variants needed in the parser for this feature.

---

## Name Resolution

Walk `ShellExpr::parts`. For each `ShellPart::Interpolated(expr)`, resolve `expr`
as a normal Ferric expression in the current scope. `ShellPart::Literal` segments
are ignored by the resolver.

---

## Type Checker

For each `ShellPart::Interpolated(expr)`:

- The expression must resolve to `Str` or `Int`. `Int` is accepted and will be
  coerced to its string representation at runtime.
- Any other type is `TypeError::ShellInterpType { found: Ty, span: Span }`.

The overall type of a `ShellExpr` is `ShellOutput`.

```rust
// Add to TypeError:
TypeError::ShellInterpType { found: Ty, span: Span }
```

Carries `Span` (Rule 5).

---

## VM (TreeWalker)

To evaluate a `ShellExpr`:

1. For each part in order:
   - `Literal(s)` — append `s` to the command string.
   - `Interpolated(expr)` — evaluate `expr`. If `Value::Str(s)`, append `s`.
     If `Value::Int(n)`, append `n.to_string()`. Other types are unreachable
     (the type checker has already rejected them).
2. Spawn a subprocess: `sh -c <cmd>` on Unix, `cmd /C <cmd>` on Windows.
3. Capture stdout. Wait for the process to exit.
4. Return `Value::new_shell_output(stdout, exit_code)`.

If the host environment does not support subprocess execution (e.g. WASM target),
return `Value::new_shell_output("", 126)`. Exit code 126 is the conventional
"command not executable" code.

### Rule 7 compliance

`Value::ShellOutput` must not be constructed directly outside `ferric_vm`:

```rust
// ILLEGAL outside ferric_vm
let v = Value::ShellOutput { stdout: "...".into(), exit_code: 0 };

// LEGAL everywhere
let v = Value::new_shell_output("...", 0);
```

Add `Value::new_shell_output(stdout: String, exit_code: i32) -> Value` to `ferric_vm`.

## Stdlib

Add field accessors for `ShellOutput`. These are the only way to read shell output
outside of `ferric_vm`:

```rust
// Register in NativeRegistry:
shell_stdout(output: ShellOutput) -> Str      // returns output.stdout
shell_exit_code(output: ShellOutput) -> Int   // returns output.exit_code
```

In practice, `.stdout` and `.exit_code` will likely become field-access syntax on
the `ShellOutput` struct value — however that field access mechanism is implemented
in M2.5. If field access on struct-like values is not yet in the language at this
point, register these as stdlib functions and update when field access arrives in M4.

---

## Async compatibility note

`$` expressions are synchronous in this task — the subprocess blocks the interpreter
thread. This is acceptable for the TreeWalker. In the async milestone (post-M3),
`$` expressions will be lowered by the compiler to awaitable operations. The
`ShellExpr` AST node in `ferric_common` requires no changes at that point — the
change is entirely in the compiler and VM.

Do not store any non-`Send` state in `ShellOutput` or `ShellExpr`. Both types must
be `Send + Sync`.
