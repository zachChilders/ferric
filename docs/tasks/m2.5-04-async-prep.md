# M2.5 — Task 4: AST Public Surface + Async Compatibility Gate

> **Prerequisite:** Task 1 (Named Parameters) must be complete before starting this
> task, since it finalises the shape of `CallExpr` and `Param` in `ferric_common`.
> Tasks 2 and 3 do not need to be complete — this task does not depend on `RequireStmt`
> or `ShellExpr` being present, but if they are already merged, their types will
> automatically be covered by the serialisation work here.

---

## What this task does

Two things that belong together because they both touch `ferric_common` globally
rather than adding a single feature:

1. **AST public surface** — makes all AST types in `ferric_common` serialisable so
   external tools (language servers, formatters, linters) can consume the AST without
   depending on parser internals.

2. **Async compatibility gate** — audits `ferric_common` and `ferric_vm` for design
   decisions that would make adding `async`/`await` later expensive, and either fixes
   them now or documents the known upgrade path.

Neither of these adds user-visible language features. Both are architectural hygiene
that is cheapest to do before M3 adds two full stage replacements.

## Part A — AST Public Surface

### Goal

External tools must be able to:
- Receive the full parsed AST as a stable, versioned data format
- Consume it without importing `ferric_parser` or any other stage crate
- Round-trip it (serialise → deserialise → same structure)

The mechanism is `serde` on all public types in `ferric_common`, plus a `--dump-ast`
CLI flag and two helper functions.

### ferric_common — changes required

Add `serde` as a dependency:

```toml
# ferric_common/Cargo.toml
[dependencies]
serde      = { version = "1", features = ["derive"] }
serde_json = "1"
```

Add `#[derive(Serialize, Deserialize)]` to every public type. This includes all
existing types (`Span`, `NodeId`, `Symbol`, `DefId`, `Interner`, `Token`, `Item`,
`Expr`, `Stmt`, `Param`, `TypeExpr`, `LexResult`, `ParseResult`, `ResolveResult`,
`TypeResult`, and all error types) as well as the new types added in Tasks 1–3
(`NamedArg`, `RequireStmt`, `RequireMode`, `ShellExpr`, `ShellPart`, `ShellOutput`).

Every type that derives `Serialize + Deserialize` must also derive `Debug + Clone`
if it doesn't already.

Add two helper functions at the crate root:

```rust
// ferric_common/src/lib.rs
pub fn ast_to_json(ast: &ParseResult) -> Result<String, serde_json::Error> {
    serde_json::to_string_pretty(ast)
}

pub fn ast_from_json(s: &str) -> Result<ParseResult, serde_json::Error> {
    serde_json::from_str(s)
}
```

### ferric_parser — one addition

Add a convenience wrapper alongside the existing `parse()` entry point:

```rust
// ferric_parser/src/lib.rs — add, do not modify existing parse()
pub fn parse_to_json(lex: &LexResult) -> Result<String, serde_json::Error> {
    let result = parse(lex);
    ferric_common::ast_to_json(&result)
}
```

This is the only addition to `ferric_parser`'s public surface. Everything else
remains private.

### CLI flag

Add `--dump-ast` to the CLI in `main.rs`:

```
ferric --dump-ast <file>
```

Runs the lexer and parser, prints the AST as pretty-printed JSON to stdout, then
exits. Does not run the resolver, type checker, or VM.

```rust
// main.rs — rough shape
if args.dump_ast {
    let lex_result  = lex(&source, &mut interner);
    let parse_result = parse(&lex_result);
    println!("{}", ferric_common::ast_to_json(&parse_result)?);
    return Ok(());
}
```

The flag lives entirely in `main.rs`. No stage is modified.

### What external tools must do and must not do

**Must do:**
- Import `ferric_common` only
- Call `ferric --dump-ast <file>` via subprocess to get the AST
- Deserialise using `ast_from_json`

**Must not do:**
- Import `ferric_parser`, `ferric_resolve`, `ferric_typecheck`, or any stage crate
- Depend on the internal structure of any stage — only `ferric_common` types are stable

If a tool needs resolved names or inferred types in the future, those will be
exported through `ResolveResult` and `TypeResult` in `ferric_common` in the same
way — as serialisable structs. The pattern established here is the template.

### Round-trip requirement

```rust
// This must hold for any valid ParseResult:
let json    = ferric_common::ast_to_json(&result).unwrap();
let result2 = ferric_common::ast_from_json(&json).unwrap();
assert_eq!(result, result2);
```

Add this as a test in `ferric_common`. It requires `PartialEq` on `ParseResult`
and all its constituent types — add `#[derive(PartialEq)]` alongside
`Serialize + Deserialize` on all public types.

## Part B — Async Compatibility Gate

This task does not implement async/await. It ensures that no decision made in M2.5
makes adding async/await later a rewrite rather than an additive pass.

### Constraint 1 — All ferric_common types must be Send + Sync

No type in `ferric_common` may contain `Rc`, `RefCell`, `*const T`, `*mut T`, or
any other non-`Send` data. Future async runtimes require that all data crossing
`.await` points implements `Send`.

**Action:** Add a compile-time assertion in `ferric_common/src/lib.rs`:

```rust
// Fails to compile if any public type is not Send + Sync
fn _assert_send_sync() {
    fn check<T: Send + Sync>() {}
    check::<Span>();
    check::<NodeId>();
    check::<Symbol>();
    check::<DefId>();
    check::<LexResult>();
    check::<ParseResult>();
    check::<ResolveResult>();
    check::<TypeResult>();
    check::<ShellOutput>();
    // add new types here as they are introduced
}
```

This function is never called — it only needs to type-check. If any type fails
`Send + Sync`, the build fails with a clear error pointing at the offending type.

### Constraint 2 — NativeRegistry upgrade path must be documented

Currently `NativeRegistry` stores synchronous function pointers:

```rust
Box<dyn Fn(...) -> Value>
```

This works for the TreeWalker. When async is added, native functions will need to
return futures. The signature will become something like:

```rust
Box<dyn Fn(...) -> Pin<Box<dyn Future<Output = Value> + Send>>>
```

**Action:** Add a comment in `ferric_stdlib/src/lib.rs` at the `NativeRegistry`
definition:

```rust
// ASYNC UPGRADE PATH: When async/await is added (post-M3), the fn type stored
// here changes to:
//   Box<dyn Fn(...) -> Pin<Box<dyn Future<Output = Value> + Send>>>
// This is a breaking change to NativeRegistry's internal type, but the public
// stage signature (NativeRegistry passed into Executor::run) does not change.
// All native function registrations will need updating at that point.
```

Do not change the type now. Just document it.

### Constraint 3 — Value variants must remain Send

All `Value` variants in `ferric_vm` must be `Send`. `Value::ShellOutput` (added in
Task 3) satisfies this — `String` and `i32` are both `Send`.

**Action:** Add the same compile-time assertion pattern to `ferric_vm/src/lib.rs`:

```rust
fn _assert_value_send() {
    fn check<T: Send>() {}
    check::<Value>();
}
```

Add a comment at the `Value` enum:

```rust
// INVARIANT: Value must remain Send. Do not add variants containing Rc, RefCell,
// raw pointers, or other non-Send types. This is required for the async upgrade path.
```

### Constraint 4 — TreeWalker frame stack must not assume a single synchronous chain

The TreeWalker's call stack is a `Vec<Frame>` on the Rust call stack. This is
fine for synchronous execution.

When the bytecode VM arrives in M3, frames move to a heap-allocated `Vec`. That
heap stack can be suspended and resumed, which is the primitive async execution
needs.

**Action:** No code change required. Add a comment in `ferric_vm/src/tree_walker.rs`:

```rust
// NOTE: The frame stack here is on the Rust call stack (recursive evaluation).
// This is intentional for the TreeWalker — it will be replaced wholesale in M3
// with a BytecodeVM whose frame stack is heap-allocated. The heap stack is what
// makes async suspension possible. Do not introduce design here that assumes
// this call chain cannot be interrupted — the M3 replacement will not have
// that assumption.
```

### ASYNC_COMPAT.md

Create `ferric_vm/ASYNC_COMPAT.md` as a living document summarising all four
constraints and their current status:

```markdown
# ferric_vm — Async Compatibility Notes

These constraints must be maintained across all milestones so that adding
async/await is an additive pass on the compiler and VM, not a rewrite.

| Constraint | Status | Action needed at async milestone |
|---|---|---|
| ferric_common types are Send + Sync | Enforced by compile-time check | None |
| NativeRegistry fn type is sync | Documented, intentional | Update fn type to return Future |
| Value variants are Send | Enforced by compile-time check | None |
| Frame stack is heap-allocated (M3+) | TreeWalker uses Rust stack (acceptable) | BytecodeVM uses heap Vec — done in M3 |
```
